# research-to-site

**Software factory workflow module 1** — Pi subagent pipeline that turns deep research into
a production Astro/bun/Vite site with teaching-forward SVG illustrations.

## Quick start

Run `factory init` once per machine, then use the scoped slash commands from any pi session
in any project:

```bash
factory init research-to-site
# or from the module's git remote:
factory init git@github.com:tenarches/research-to-site.git

# then, from anywhere:
/rts-research <topic>
```

`factory init` initializes the workspace git repo (`$DEEP_RESEARCH_WORKSPACE`), installs the
module's subagents and slash commands globally, and registers required env. Afterwards
`/rts-research`, `/rts-build-site`, `/rts-list-runs`, and `/factory-status` work from any
pi session — you do not need to `cd` into this directory.

> **Development mode:** to iterate on the module itself, `cd` into this directory and run
> `pi` so the project-scoped `.pi/subagents/` and `.pi/prompts/` are picked up directly.

## Architecture

```
pi session (this directory)
  └─ pi subagent: research-to-site  ← orchestrator
       └─ pi subagent: team-lead    ← tmux coordinator + git rollup
            └─ tmux session: run_<ts>
                 ├─ window: researcher   → worktree, pi --prompt
                 ├─ window: svg-<slug>   → worktree, pi --prompt (N parallel)
                 └─ window: site-builder → worktree, pi --prompt
```

Workers are isolated pi agent processes. The team lead monitors each window, sends
targeted corrective prompts on stall/failure, and merges all worktrees on completion.

### Workspace as a git repo (rollup)

`$DEEP_RESEARCH_WORKSPACE` is itself a git repository, created by `factory init`. Each worker
runs in a git **worktree** of that repo:

- The team lead creates an integration branch from `main`, then one worktree/branch per worker.
- Before merging a completed worker, the lead commits any output the worker left uncommitted
  in its worktree.
- Worker branches are merged sequentially into the integration branch with `--no-ff` (in launch
  order; failed/killed branches are skipped).
- Finally the integration branch is merged into `main`, so every completed run lands on `main`.

Built sites can additionally be delivered outside the repo via the `| deliver to: <dir>` suffix
(or the `$FACTORY_DELIVER_DIR` default), which copies `site/dist/` to `<dir>/<run_id>/`.

## Setup

### 1. Install prerequisites

```bash
pi install npm:@piotr-oles/pi-subagents
# Ensure qwen3.6-35b-a3b is registered in your pi model config
# Ensure bun and tmux are installed on the target machine
```

### 2. Configure the global env file

```bash
mkdir -p ~/.deep-research-agent
cp .env.example ~/.deep-research-agent/.env
# Edit ~/.deep-research-agent/.env and fill in all values
```

This file is the config anchor — **every subagent sources it from that fixed path**,
regardless of which worktree or sandbox it runs in.

### 3. (Optional) Make subagents globally available

```bash
mkdir -p ~/.pi/agent/subagents
cp .pi/subagents/*.md ~/.pi/agent/subagents/
# Or symlink: ln -s $(pwd)/.pi/subagents/*.md ~/.pi/agent/subagents/
```

## Usage

Start a pi session from this directory (for project-scoped subagents and slash commands):

```bash
cd /path/to/software-factory/research-to-site
pi
```

### Slash commands

| Command | Effect |
|---------|--------|
| `/rts-research <topic> [\| deliver to: <dir>]` | Full pipeline: research → SVG illustrations → Astro site |
| `/rts-build-site [run_dir] [\| deliver to: <dir>]` | Build site for a specific run, or pick up the latest pending run |
| `/rts-list-runs` | Table of all research runs with site build status |
| `/factory-status` | Cross-module factory status: runs, active tmux sessions, available modules |

### Manual invocation inside pi

```
# Full pipeline
Research and build site for: <topic>

# Research only (no tmux, no site build)
deep-researcher: <topic>

# Illustrate one concept
svg-illustrator: Concept: <phrase> | Type: diagram | Caption: <text> | Output path: /absolute/path.svg

# Build site from an existing run (optional delivery)
site-builder: Build site for: /absolute/path/to/run_dir | deliver to: /absolute/dir

# Team lead only (with a pre-built worker spec)
team-lead: Lead pipeline for: <run_dir> | run_id: ... | workers: [...]
```

## Output layout

```
~/.deep-research-agent/
  .env                          ← global config anchor
  runs.json                     ← run registry (appended after each research run)
  workspace/
    run_<timestamp>/
      manifest.json             ← OKF document index + metadata
      final_report.md           ← research summary
      doc_*.md                  ← source documents (OKF frontmatter)
      svgs/
        <concept-slug>.svg      ← one per concept, generated by svg-illustrator
      site/
        (Astro project)
        dist/                   ← final static output
          index.html
          ...
      wt-researcher/            ← git worktree (cleaned up after merge)
      wt-svg-<slug>/            ← git worktrees (cleaned up after merge)
      wt-site/                  ← git worktree (cleaned up after merge)
```

### Dev server

```bash
cd ~/.deep-research-agent/workspace/run_<ts>/site
bun run dev
```

## Adding a new workflow module

This module is designed to be the first in a composable software factory. To add another:

1. Create `software-factory/<new-module>/` with `.pi/subagents/` and `.pi/prompts/`.
2. Symlink or copy `team-lead.md` — the tmux coordination pattern is generic and reusable.
3. Define your worker and orchestrator subagent specs.
4. Add a module-scoped prompt template (e.g. `/<module>-run`) under `.pi/prompts/`.
5. `/factory-status` auto-discovers modules with `.pi/subagents/` directories.

## Pi subagents concurrency

```bash
PI_SUBAGENTS_MAX_CONCURRENT=4 pi
```
